Pas Actor Actiune Sistem
1 Client Completeaza formularul multi-step (locatii, tip, servicii, data, contact) HomeRequestForm → /api/requests/createGuest
2 Sistem Genereaza cod secvential REQ-XXXXXX, leaga la cont existent daca emailul se potriveste Firestore meta/counters + requests
3 Sistem Notifica companiile (Firestore notifications + email Resend) Toate companiile primesc notificare
4 Client Incarca poze/video prin link tokenizat (7 zile expiry) - optional upload/[token] → Firebase Storage
5 Companie Vede cererea in RequestsView, trimite oferta (pret + mesaj) Tranzactie Firestore: -credite + creare oferta atomic
6 Client Vede ofertele, accepta sau respinge /api/offers/accept sau /decline
7 Sistem La accept: partajeaza datele de contact ale clientului cu compania Email detaliat catre companie
8 Ambii Comunicare via chat real-time ChatWindow + Firestore subcollection
9 Sistem Auto-refund credite daca clientul nu raspunde 72h /api/cron/auto-refund
10 Client Lasa recenzie dupa serviciu /reviews/new → Firestore reviews

✅ Sprint 5 — Hardening (8/8 rezolvate)

Al doilea audit — securitate, XSS, cleanup, si best practices.

REZOLVAT36. .env.production comis in git
.gitignore + git rm --cached

Fisierul cu TOATE secretele (Firebase Admin private key, Resend API key, CRON_API_KEY) era tracked in git. Adaugat in .gitignore si eliminat din index.

→ ⚠️ ROTESTE toate secretele — raman in git history

REZOLVAT37. Logger silentios in productie
utils/logger.ts

error() si warn() erau protejate de if(isDevelopment) — erorile in productie erau complet pierdute.

→ error() si warn() logheaza mereu. PM2 capteaza stderr.

REZOLVAT38. sendUploadReminders auth fail-open
pages/api/sendUploadReminders.ts

Auth se activa doar daca process.env.CRON_API_KEY exista. Fara variabila, endpoint-ul era complet deschis.

→ Fail-closed: daca CRON_API_KEY lipseste, returneaza 401.

REZOLVAT39. sharp in devDependencies
package.json

Next.js Image Optimization necesita sharp in productie. Era doar in devDependencies.

→ Mutat in dependencies.

REZOLVAT40. createGuest face self-fetch circular
pages/api/requests/createGuest.ts

Trimitea emailuri prin fetch() catre /api/send-email pe acelasi server. Overhead, latenta, risc timeout.

→ Import direct sendEmail() + emailTemplates din services/email.ts.

REZOLVAT41. XSS in template-uri email
services/email.ts, pages/api/send-email.ts

Toate cele 9+ template-uri interpoleaza date user fara sanitizare. Atacator putea injecta HTML/script in emailuri.

→ Adaugat escapeHtml() pe toate inputurile user in fiecare template.

REZOLVAT42. Client-side fallback bypasseaza server
components/home/HomeRequestForm.tsx

Pe status 503, formularul apela createGuestRequest() direct din client, ocolind rate limiting si validarea server.

→ Eliminat fallback. Doar calea API server-side ramane.

REZOLVAT43-44. Cleanup cod mort
next.config.js, generateUploadLink.ts, GuaranteeSection_old.tsx.bak

Fisier .bak stale sters, lucide-react din modularizeImports (pachet neinstalat), comentariu EmailJS stale actualizat, @deprecated pe createGuestRequest().

✅ Probleme Critice — TOATE REZOLVATE (5/5)

Toate problemele critice au fost rezolvate in sesiunea de audit din 9 februarie 2026.

REZOLVAT1. Formularul de contact NU trimite emailuri
pages/contact.tsx → handleSubmit()

handleSubmit face doar setTimeout(resolve, 1500) — simuleaza trimiterea. Mesajele clientilor sunt pierdute complet.

→ Implementeaza apel catre /api/send-email cu tipul contactForm

REZOLVAT2. Firebase Storage — upload-uri neautentificate permise
firebase.storage.rules (linia ~37)

Regula write include conditii OR fara request.auth != null. Oricine poate incarca fisiere pe server.

→ Adauga request.auth != null in toate conditiile de write

REZOLVAT3. API /api/send-email fara autentificare
pages/api/send-email.ts

Orice cerere POST e acceptata. Un atacator poate trimite emailuri arbitrare prin contul Resend.

→ Adauga API key secret sau verificare Firebase token

REZOLVAT4. API /api/generateUploadLink fara autentificare
pages/api/generateUploadLink.ts

Oricine poate genera token-uri de upload pentru orice cerere.

→ Adauga verifyAuth(req) si verificare ownership

REZOLVAT5. console.error in auto-refund expune date sensibile
pages/api/cron/auto-refund.ts (liniile 145, 168)

console.error direct care logheaza in productie, potential expunand IDs si erori Firestore.

→ Inlocuieste cu logger.error din utils/logger.ts

🟠 Prioritate Ridicata (3)

HIGHT2. generateUploadLink — auth doar pe email
pages/api/generateUploadLink.ts

Auth-ul verifica doar daca email-ul din body se potriveste cu un request existent. Nu verifica Firebase token sau ownership real. Un atacator care stie un email valid poate genera tokeni de upload.

→ Adauga verifyIdToken() sau cel putin un request-specific token/secret pe langa email match.

HIGHT3. Firestore rules uploadTokens — lipsa guard pe read
firebase.firestore.rules

Oricine autentificat poate citi orice token din colectia uploadTokens, inclusiv tokeni folositi. Ar trebui sa limiteze read-ul la used == false sau la owner.

→ Adauga resource.data.used == false in regula de read, sau restrictioneaza pe baza emailului utilizatorului.

HIGHT4. INTERNAL_API_SECRET — fara validare la boot
pages/api/send-email.ts, lib/apiAuth.ts

Daca INTERNAL_API_SECRET lipseste sau e prea scurt, API-ul accepta orice request intern fara avertisment. Ar trebui sa refuze pornirea sau sa logheze un avertisment critic.

→ Adauga validare la import: daca lipseste sau are <32 caractere, logheaza logCritical() si intoarce 500 pe fiecare request.
